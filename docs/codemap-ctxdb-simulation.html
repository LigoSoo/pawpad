codemap · ctxdb 는 어떻게 토큰을 줄이나

파일을 전부 열지 않고 레지스트리 1줄로 위치를 찾고, 세션 시작 시 키워드로 필요한 것만 로드한다. 실제 프로젝트 TodayQuest(Flutter, lib 160파일) 수치로 시뮬레이션.

PawPad Toolkit · 실측 데이터 기반 인터랙티브 데모
BUG
실제 리포트(v2.41 계기): "작은 버그 하나 고치는데 context가 2~3배로 불어나고 토큰이 급증". 원인 — 에이전트가 codemap을 안 거치고 소스 전체를 뒤져서 대상을 찾았다.

아래 시뮬레이션은 같은 작업을 codemap 없이(full-scan) vs codemap 경유로 나란히 실행해, 대상을 찾기까지 읽는 토큰을 실시간으로 센다.

Act 1 · 위치 찾기

codemap — 심볼 위치 레지스트리

도메인:심볼 → 파일경로 + 시그니처 + 역할(1줄). 파일을 열지 않고 어디에 뭐가 있는지 1줄로 안다.

TARGET
작업: "퀘스트 완료 토글을 눌러도 보상이 안 들어온다" 버그 수정 → 실제 고칠 곳은 quest_providers.dart 의 CompletionController.toggle 단 하나. 문제는 에이전트가 그 위치를 모른다는 것.

codemap 전체는 60KB(~15K tok)지만 통째로 읽지 않는다. trim-router가 _root.md(2KB)→keywords.md→features/{id}.md(각 4KB cap)로 쪼개, 매 조회는 3파일(~1.9K tok)만 경유한다.

Act 2 · 세션 시작

ctxdb — 키워드 depth 컨텍스트 로더

세션을 다시 열 때 과거 이력을 전부 다시 읽지 않는다. INDEX → L1 → L2 를 첫 메시지 키워드로 매칭해 최소한만 로드.

RESTART
새 세션 시작. 이전 작업은 "펫 부화 UX". 필요한 건 지금 상태 요약(L2) 뿐인데, 순진하게 하면 지난 달 구현 로그 전체(L3, 2.6K tok)까지 다 읽는다.

원리

왜 줄어드나 — 한 줄 요약

codemap

파일 read ≈ 13 tok/줄. 150줄 파일 하나가 ~2K tok. 위치를 모르면 후보 파일을 여러 개 열어야 하고, 그게 곧 context 폭증이다.

codemap은 그 "어디에 뭐가 있나"를 심볼당 1줄로 미리 적어둔다. 시그니처·역할까지 있어 파일을 안 열고도 고칠 지점을 확정 → 딱 1파일만 open.

trim-router: 큰 codemap을 _root+keywords+features/{id}로 분할해 "통째 읽기" 사고 자체를 봉쇄.

ctxdb

세션 재개 비용은 "과거를 얼마나 다시 읽느냐"에 비례. 콜드 재개는 CLAUDE.md·이력·설계를 통째로 흡수한다.

ctxdb는 이력을 depth로 층화(INDEX→L1→L2→L3)하고, 첫 메시지 키워드로 필요한 깊이까지만 로드한다. 지난 달 상세 로그(L3)는 건드리지 않는다.

UserPromptSubmit hook이 이 최소 로드를 자동 수행 → 에이전트 부담은 더 줄어든다.

두 축은 관측(v2.41)과도 맞물린다: 응답의 📡 Retrieval 선언 + statusline cmap/ctx/src 실측 카운터가 "codemap을 안 거치고 full-scan했는지"를 실시간으로 드러내, 위 시나리오의 이탈을 즉시 발견한다.
